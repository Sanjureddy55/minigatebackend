================================================================================
  MINIGATE BACKEND — COMPLETE API REFERENCE
  For Frontend Integration Team
  Base URL : http://127.0.0.1:8000          (dev)
  Base URL : https://api.minigate.in        (prod — update when deployed)
  Auth     : Bearer Token (JWT)
  OTP      : Always 123456 in dev/staging
================================================================================

HOW TO USE THIS DOCUMENT
─────────────────────────
1. Every request that needs auth must include:
   Header → Authorization: Bearer <access_token>

2. Tokens are returned on login. The access token expires in 60 min.
   Refresh it using POST /api/accounts/token/refresh/.

3. Roles and what they can access:
   super-admin     → Platform Admin APIs  (/api/platform-admin/*)
   society-admin   → Society Admin APIs   (/api/society-admin/*)
   accountant      → Accountant APIs      (/api/accountant/*)
   resident        → Resident APIs        (/api/resident/*)
   security-guard  → Guard APIs           (/api/security-guard/*)

4. Errors always follow this shape:
   { "success": false, "message": "Human readable error." }

5. Validation errors follow this shape:
   { "field_name": ["Error message."] }

6. Share this file with Claude AI and say:
   "This is my backend API doc. Help me integrate [endpoint name]."

6.9  DELIVERY VERIFICATION  (OTP-based handoff workflow)
────────────────────────────────────────────────────────────────────────────────
Base : /api/security-guard/delivery-verify/
Auth : security-guard

Workflow:
  1. Delivery arrives at gate  → Guard creates entry (POST /)
  2. Guard approves            → OTP sent to resident (POST /<pk>/approve/)
  3. Resident shares OTP       → Guard enters it       (POST /<pk>/verify-otp/)
  4. System marks COLLECTED automatically on OTP success

Status lifecycle:
  pending → approved / at_gate → otp_sent → collected
                              ↘ rejected / returned

LIST
  GET /api/security-guard/delivery-verify/
  Params: status, flat, date, search, page
  Response 200: { "count": N, "results": [ <DeliveryEntry>, ... ] }

PENDING (awaiting guard action)
  GET /api/security-guard/delivery-verify/pending/
  Response 200: { "count": N, "results": [ ... ] }

AT GATE (arrived, not yet collected)
  GET /api/security-guard/delivery-verify/at-gate/

SUMMARY
  GET /api/security-guard/delivery-verify/summary/
  Response 200:
  {
    "success": true,
    "data": {
      "today_total": 8,

  Response 200: { "success": true, "status": "rejected" }

================================================================================
  SECTION 8 — INTEGRATION QUICKSTART
  (Copy-paste for your frontend / Claude AI)
================================================================================

STEP 1 — LOGIN AND SAVE TOKENS
  POST /api/accounts/login/mobile/
  Body: { "mobile": "9000000001", "otp_code": "123456" }
  → Save tokens.access and tokens.refresh in localStorage/SecureStorage

STEP 2 — USE TOKEN ON EVERY REQUEST
  Headers: { "Authorization": "Bearer <access_token>" }

STEP 3 — REFRESH WHEN EXPIRED
  POST /api/accounts/token/refresh/
  Body: { "refresh": "<refresh_token>" }
  → Replace old access with new access

STEP 4 — REDIRECT BY ROLE AFTER LOGIN
  Read data.role.slug from login response:
    "super-admin"    → go to Platform Admin dashboard
    "society-admin"  → go to Society Admin dashboard
    "accountant"     → go to Accountant dashboard
    "resident"       → go to Resident dashboard

STEP 5 — READ FEATURES FOR PERMISSION GATING
  Login response has data.features[] — each item has:
    module, can_view, can_create, can_edit, can_delete
  Use these to show/hide UI elements.
